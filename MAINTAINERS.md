# Maintainers

This document lists the maintainers of the Ancient Nerds Map project and describes our governance model.

## Current Maintainers

| Name | GitHub | Role | Areas of Focus |
|------|--------|------|----------------|
| Ancient Nerds Team | [@AncientNerds](https://github.com/AncientNerds) | Lead Maintainer | All areas |

## Roles

### Lead Maintainer
- Final decision authority on project direction
- Manages releases and versioning
- Reviews and merges PRs to protected branches
- Handles security issues

### Maintainer
- Reviews and approves PRs
- Triages issues
- Contributes to roadmap discussions
- Mentors contributors

### Contributor
- Submits PRs and issues
- Participates in discussions
- Helps with documentation and testing

## Governance Model

### Decision Making

1. **Minor changes** (bug fixes, small features): Any maintainer can approve and merge
2. **Major changes** (architecture, breaking changes): Requires lead maintainer approval
3. **Controversial changes**: Discussed in GitHub Discussions, decided by maintainer consensus

### Code Review Process

1. All PRs require at least one maintainer approval
2. PRs affecting security require lead maintainer review
3. CI must pass before merging
4. Maintainers should respond to PRs within 7 days

### Release Process

1. Releases follow [Semantic Versioning](https://semver.org/)
2. Releases are tagged and documented in CHANGELOG.md
3. Major releases require lead maintainer approval
4. Security patches are released as soon as possible

## Becoming a Maintainer

Contributors who demonstrate:
- Consistent, high-quality contributions
- Understanding of project goals and architecture
- Helpful participation in issues and discussions
- Adherence to code of conduct

May be invited to become maintainers.

## Communication

- **GitHub Issues**: Bug reports, feature requests
- **GitHub Discussions**: General questions, ideas, RFC discussions
- **Email**: ancientnerds@proton.me (security issues, private matters)

## Security
